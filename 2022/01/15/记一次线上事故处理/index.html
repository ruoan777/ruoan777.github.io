前言

在一个愉快的周四，正在愉快的喝着咖啡敲bug，突然TS事件群里有人艾特，得，又有锅要来了。

事情的原因是这样的，线上某个订单的交易快照突然打不开了，文后贴上咱们的交易快照的存储流程(已脱敏)。
具体定位过程真是一波三折，最终的原因是一个并发的问题(但凡是并发问题，好像一般都很难定位-。-)

虽然最终问题代码不是我写的，是在某一次新增功能后引入的，但整个排查过程很有意思，也获得了不少经验，值得记录下来。

问题原因

下面是问题代码(公司代码没法直接贴过来，下面是对问题进行抽象后的大致代码框架)

@Controller
public class Controllers {

    @Autowired 
    private Process process;

    public void saveSnapShot() {
        Executors.newSingleThreadExecutor().submit(() -> {
            process.setVersion(request.getVersion());
            process.doSave();
        });
    }
}

@Service
public class Process {
    
    private Integer version = 0;
    
    public void setVersion(Integer version) {
        this.version = version;
    }

    public void doSave(
        if (version == 1) {
            doSomeThingWhenVersion1();
            return;
        }
        if (version == 2) {
            doSomeThingWhenVersion2();
            return;
        }
    }
}

作者当初估计是想在版本号不同的时候走不同的存储逻辑，但这里的代码犯了三个错误。

  • 错误1:单例类里使用实例变量，会有线程安全问题
    假如第一个线程进来将 version 设置为1，这时候准备走 doSave() 方法，结果另一个线程这时也恰巧进来了，将 version 设置为2，那么第一个线程估计有点懵逼，它一判断 version == 2，咣当就走到第二个if进去了。
    让doSomeThingWhenVersion2()去执行version=1的逻辑处理，报错就难免了，这就是我们这次事件的本质原因。

有的小朋友可能就会问了，为啥要设置一个实例变量，而不是通过传参的形式给doSave()方法嘞？

说来话长，实际的生产代码比上面这个复杂多了，写它的时候可能没有注意到，不小心就犯了这么一个本质上很简单的错误。

  • 错误2:在方法的执行路径上创建线程池
    这样写和直接new Thread()有啥区别嘞？

  • 错误3:线程池里执行submit()，会将里面的异常吞掉
    这就是本次问题定位异常困难的原因，如果这里有异常记录，有错误的堆栈，那不是分分钟就找到问题了？

问题修复

找到了原因后，问题就很好修复了，线程池的创建改一改，不再使用实例变量的形式，改用传参的形式即可。

后记
